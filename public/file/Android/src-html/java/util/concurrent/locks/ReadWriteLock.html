007    package java.util.concurrent.locks;
008    
009    /**
010     * A <tt>ReadWriteLock</tt> maintains a pair of associated {@link
011     * Lock locks}, one for read-only operations and one for writing.
012     * The {@link #readLock read lock} may be held simultaneously by
013     * multiple reader threads, so long as there are no writers.  The
014     * {@link #writeLock write lock} is exclusive.
015     * 
016     * <p>A read-write lock allows for a greater level of concurrency in
017     * accessing shared data, than that permitted by a mutual exclusion lock.
018     * It exploits the fact that while only a single thread at a time (a
019     * <em>writer</em> thread) can modify the shared data, in many cases any 
020     * number of threads can concurrently read the data (hence <em>reader</em>
021     * threads).
022     * In theory, the increase in concurrency permitted by the use of a read-write
023     * lock will lead to performance improvements over the use of a mutual
024     * exclusion lock. In practice this increase in concurrency will only be fully
025     * realized on a multi-processor, and then only if the access patterns for
026     * the shared data are suitable.
027     *
028     * <p>Whether or not a read-write lock will improve performance over the use
029     * of a mutual exclusion lock depends on the frequency that the data is
030     * read compared to being modified, the duration of the read and write 
031     * operations, and the contention for the data - that is, the number of
032     * threads that will try to read or write the data at the same time.
033     * For example, a collection that is initially populated with data and
034     * thereafter infrequently modified, while being frequently searched
035     * (such as a directory of some kind) is an ideal candidate for the use of
036     * a read-write lock. However, if updates become frequent then the data
037     * spends most of its time being exclusively locked and there is little, if any
038     * increase in concurrency. Further, if the read operations are too short
039     * the overhead of the read-write lock implementation (which is inherently
040     * more complex than a mutual exclusion lock) can dominate the execution
041     * cost, particularly as many read-write lock implementations still serialize
042     * all threads through a small section of code. Ultimately, only profiling
043     * and measurement will establish whether the use of a read-write lock is
044     * suitable for your application.
045     *
046     *
047     * <p>Although the basic operation of a read-write lock is straight-forward,
048     * there are many policy decisions that an implementation must make, which
049     * may affect the effectiveness of the read-write lock in a given application.
050     * Examples of these policies include:
051     * <ul>
052     * <li>Determining whether to grant the read lock or the write lock, when
053     * both readers and writers are waiting, at the time that a writer releases
054     * the write lock. Writer preference is common, as writes are expected to be
055     * short and infrequent. Reader preference is less common as it can lead to
056     * lengthy delays for a write if the readers are frequent and long-lived as
057     * expected. Fair, or &quot;in-order&quot; implementations are also possible.
058     *
059     * <li>Determining whether readers that request the read lock while a 
060     * reader is active and a writer is waiting, are granted the read lock.
061     * Preference to the reader can delay the writer indefinitely, while
062     * preference to the write can reduce the potential for concurrency.
063     *
064     * <li>Determining whether the locks are reentrant: can a thread with the
065     * write lock reacquire it? can it acquire a read lock while holding the
066     * write lock? is the read lock itself reentrant?
067     *
068     * <li>Can the write lock be downgraded to a read lock without allowing
069     * an intervening writer? Can a read lock be upgraded to a write lock,
070     * in preference to other waiting readers or writers?
071     *
072     * </ul>
073     * You should consider all of these things when evaluating the suitability
074     * of a given implementation for your application.
075     *
076     * @see ReentrantReadWriteLock
077     * @see Lock
078     * @see ReentrantLock
079     *
080     * @since 1.5
081     * @author Doug Lea
082     */
083    public interface ReadWriteLock {
084        /**
085         * Returns the lock used for reading.
086         *
087         * @return the lock used for reading.
088         */
089        Lock readLock();
090    
091        /**
092         * Returns the lock used for writing.
093         *
094         * @return the lock used for writing.
095         */
096        Lock writeLock();
097    }
