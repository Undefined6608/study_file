036    /**
037     * A concrete implementation class for {@code Collation}.
038     * <p>
039     * {@code RuleBasedCollator} has the following restrictions for efficiency
040     * (other subclasses may be used for more complex languages):
041     * </p>
042     * <ol>
043     * <li> If a French secondary ordering is specified it applies to the whole
044     * collator object.</li>
045     * <li> All non-mentioned Unicode characters are at the end of the collation
046     * order.</li>
047     * <li> If a character is not located in the {@code RuleBasedCollator}, the
048     * default Unicode Collation Algorithm (UCA) rulebased table is automatically
049     * searched as a backup.</li>
050     * </ol>
051     * <p>
052     * The collation table is composed of a list of collation rules, where each rule
053     * is of three forms:
054     * </p>
055     * <blockquote>
056     * 
057     * <pre>
058     * <modifier>
059     * <relation> <text-argument>
060     * <reset> <text-argument>
061     * </pre>
062     * 
063     * </blockquote>
064     * <p>
065     * The rule elements are defined as follows:
066     * </p>
067     * <ul type="disc">
068     * <li><strong>Text-Argument</strong>: A text-argument is any sequence of
069     * characters, excluding special characters (that is, common whitespace
070     * characters [0009-000D, 0020] and rule syntax characters [0021-002F,
071     * 003A-0040, 005B-0060, 007B-007E]). If those characters are desired, you can
072     * put them in single quotes (for example, use '&' for ampersand). Note that
073     * unquoted white space characters are ignored; for example, {@code b c} is
074     * treated as {@code bc}.</li>
075     * <li><strong>Modifier</strong>: There is a single modifier which is used to
076     * specify that all accents (secondary differences) are backwards.
077     * <p>
078     * '@' : Indicates that accents are sorted backwards, as in French.
079     * </p>
080     * </li>
081     * <li><strong>Relation</strong>: The relations are the following:
082     * <ul type=square>
083     * <li>'<' : Greater, as a letter difference (primary)
084     * <li>';' : Greater, as an accent difference (secondary)
085     * <li>',' : Greater, as a case difference (tertiary)
086     * <li>'=' : Equal
087     * </ul>
088     * </li>
089     * <li><strong>Reset</strong>: There is a single reset which is used primarily
090     * for contractions and expansions, but which can also be used to add a
091     * modification at the end of a set of rules.
092     * <p>
093     * '&' : Indicates that the next rule follows the position to where the reset
094     * text-argument would be sorted.
095     * </p>
096     * </li>
097     * </ul>
098     * <p>
099     * This sounds more complicated than it is in practice. For example, the
100     * following are equivalent ways of expressing the same thing:
101     * </p>
102     * <blockquote>
103     * 
104     * <pre>
105     * a < b < c
106     * a < b & b < c
107     * a < c & a < b
108     * </pre>
109     * 
110     * </blockquote>
111     * <p>
112     * Notice that the order is important, as the subsequent item goes immediately
113     * after the text-argument. The following are not equivalent:
114     * </p>
115     * <blockquote>
116     * 
117     * <pre>
118     * a < b & a < c
119     * a < c & a < b
120     * </pre>
121     * 
122     * </blockquote>
123     * <p>
124     * Either the text-argument must already be present in the sequence, or some
125     * initial substring of the text-argument must be present. For example
126     * {@code "a < b & ae < e"} is valid since "a" is present in the sequence before
127     * "ae" is reset. In this latter case, "ae" is not entered and treated as a
128     * single character; instead, "e" is sorted as if it were expanded to two
129     * characters: "a" followed by an "e". This difference appears in natural
130     * languages: in traditional Spanish "ch" is treated as if it contracts to a
131     * single character (expressed as {@code "c < ch < d"}), while in traditional
132     * German a-umlaut is treated as if it expands to two characters (expressed as
133     * {@code "a,A < b,B  ... & ae;\u00e3 & AE;\u00c3"}, where \u00e3 and \u00c3
134     * are the escape sequences for a-umlaut).
135     * </p>
136     * <h4>Ignorable Characters</h4>
137     * <p>
138     * For ignorable characters, the first rule must start with a relation (the
139     * examples we have used above are really fragments; {@code "a < b"} really
140     * should be {@code "< a < b"}). If, however, the first relation is not
141     * {@code "<"}, then all text-arguments up to the first {@code "<"} are
142     * ignorable. For example, {@code ", - < a < b"} makes {@code "-"} an ignorable
143     * character.
144     * </p>
145     * <h4>Normalization and Accents</h4>
146     * <p>
147     * {@code RuleBasedCollator} automatically processes its rule table to include
148     * both pre-composed and combining-character versions of accented characters.
149     * Even if the provided rule string contains only base characters and separate
150     * combining accent characters, the pre-composed accented characters matching
151     * all canonical combinations of characters from the rule string will be entered
152     * in the table.
153     * </p>
154     * <p>
155     * This allows you to use a RuleBasedCollator to compare accented strings even
156     * when the collator is set to NO_DECOMPOSITION. However, if the strings to be
157     * collated contain combining sequences that may not be in canonical order, you
158     * should set the collator to CANONICAL_DECOMPOSITION to enable sorting of
159     * combining sequences. For more information, see <a
160     * href="http://www.aw.com/devpress">The Unicode Standard, Version 3.0</a>.
161     * </p>
162     * <h4>Errors</h4>
163     * <p>
164     * The following rules are not valid:
165     * </p>
166     * <ul type="disc">
167     * <li>A text-argument contains unquoted punctuation symbols, for example
168     * {@code "a < b-c < d"}.</li>
169     * <li>A relation or reset character is not followed by a text-argument, for
170     * example {@code "a < , b"}.</li>
171     * <li>A reset where the text-argument (or an initial substring of the
172     * text-argument) is not already in the sequence or allocated in the default UCA
173     * table, for example {@code "a < b & e < f"}.</li>
174     * </ul>
175     * <p>
176     * If you produce one of these errors, {@code RuleBasedCollator} throws a
177     * {@code ParseException}.
178     * </p>
179     * <h4>Examples</h4>
180     * <p>
181     * Normally, to create a rule-based collator object, you will use
182     * {@code Collator}'s factory method {@code getInstance}. However, to create a
183     * rule-based collator object with specialized rules tailored to your needs, you
184     * construct the {@code RuleBasedCollator} with the rules contained in a
185     * {@code String} object. For example:
186     * </p>
187     * <blockquote>
188     * 
189     * <pre>
190     * String Simple = "< a < b < c < d";
191     * 
192     * RuleBasedCollator mySimple = new RuleBasedCollator(Simple);
193     * </pre>
194     * 
195     * </blockquote>
196     * <p>
197     * Or:
198     * </p>
199     * <blockquote>
200     * 
201     * <pre>
202     * String Norwegian = "< a,A< b,B< c,C< d,D< e,E< f,F< g,G< h,H< i,I"
203     *         + "< j,J< k,K< l,L< m,M< n,N< o,O< p,P< q,Q< r,R" 
204     *         + "< s,S< t,T< u,U< v,V< w,W< x,X< y,Y< z,Z"
205     *         + "< \u00E5=a\u030A,\u00C5=A\u030A" 
206     *         + ";aa,AA< \u00E6,\u00C6< \u00F8,\u00D8";
207     * 
208     * RuleBasedCollator myNorwegian = new RuleBasedCollator(Norwegian);
209     * </pre>
210     * 
211     * </blockquote>
212     * <p>
213     * Combining {@code Collator}s is as simple as concatenating strings. Here is
214     * an example that combines two {@code Collator}s from two different locales:
215     * </p>
216     * <blockquote>
217     * 
218     * <pre>
219     * // Create an en_US Collator object
220     * RuleBasedCollator en_USCollator = (RuleBasedCollator)Collator
221     *         .getInstance(new Locale("en", "US", ""));
222     * 
223     * // Create a da_DK Collator object
224     * RuleBasedCollator da_DKCollator = (RuleBasedCollator)Collator
225     *         .getInstance(new Locale("da", "DK", ""));
226     * 
227     * // Combine the two collators
228     * // First, get the collation rules from en_USCollator
229     * String en_USRules = en_USCollator.getRules();
230     * 
231     * // Second, get the collation rules from da_DKCollator
232     * String da_DKRules = da_DKCollator.getRules();
233     * 
234     * RuleBasedCollator newCollator = new RuleBasedCollator(en_USRules + da_DKRules);
235     * // newCollator has the combined rules
236     * </pre>
237     * 
238     * </blockquote>
239     * <p>
240     * The next example shows to make changes on an existing table to create a new
241     * {@code Collator} object. For example, add {@code "& C < ch, cH, Ch, CH"} to
242     * the {@code en_USCollator} object to create your own:
243     * </p>
244     * <blockquote>
245     * 
246     * <pre>
247     * // Create a new Collator object with additional rules
248     * String addRules = "& C < ch, cH, Ch, CH";
249     * 
250     * RuleBasedCollator myCollator = new RuleBasedCollator(en_USCollator + addRules);
251     * // myCollator contains the new rules
252     * </pre>
253     * 
254     * </blockquote>
255     * <p>
256     * The following example demonstrates how to change the order of non-spacing
257     * accents:
258     * </p>
259     * <blockquote>
260     * 
261     * <pre>
262     * // old rule
263     * String oldRules = "= \u00a8 ; \u00af ; \u00bf" + "< a , A ; ae, AE ; \u00e6 , \u00c6"
264     *         + "< b , B < c, C < e, E & C < d, D";
265     * 
266     * // change the order of accent characters
267     * String addOn = "& \u00bf ; \u00af ; \u00a8;";
268     * 
269     * RuleBasedCollator myCollator = new RuleBasedCollator(oldRules + addOn);
270     * </pre>
271     * 
272     * </blockquote>
273     * <p>
274     * The last example shows how to put new primary ordering in before the default
275     * setting. For example, in the Japanese {@code Collator}, you can either sort
276     * English characters before or after Japanese characters:
277     * </p>
278     * <blockquote>
279     * 
280     * <pre>
281     * // get en_US Collator rules
282     * RuleBasedCollator en_USCollator = (RuleBasedCollator)
283     *     Collator.getInstance(Locale.US);
284     * 
285     * // add a few Japanese character to sort before English characters
286     * // suppose the last character before the first base letter 'a' in
287     * // the English collation rule is \u30A2
288     * String jaString = "& \u30A2 , \u30FC < \u30C8";
289     * 
290     * RuleBasedCollator myJapaneseCollator = 
291     *     new RuleBasedCollator(en_USCollator.getRules() + jaString);
292     * </pre>
293     * 
294     * </blockquote>
295     * 
296     * @since Android 1.0
297     */
298    public class RuleBasedCollator extends Collator {
299    
300        RuleBasedCollator(com.ibm.icu4jni.text.Collator wrapper) {
301            super(wrapper);
302        }
303    
304        /**
305         * Constructs a new instance of {@code RuleBasedCollator} using the
306         * specified {@code rules}. The {@code rules} are usually either
307         * hand-written based on the {@link RuleBasedCollator class description} or
308         * the result of a former {@link #getRules()} call.
309         * <p>
310         * Note that the {@code rules} are actually interpreted as a delta to the
311         * standard Unicode Collation Algorithm (UCA). Hence, an empty {@code rules}
312         * string results in the default UCA rules being applied. This differs
313         * slightly from other implementations which work with full {@code rules}
314         * specifications and may result in different behavior.
315         *
316         * @param rules
317         *            the collation rules.
318         * @throws NullPointerException
319         *             if {@code rules} is {@code null}.
320         * @throws ParseException
321         *             if {@code rules} contains rules with invalid collation rule
322         *             syntax.
323         * @since Android 1.0
324         */
325        public RuleBasedCollator(String rules) throws ParseException {
326            if (rules == null) {
327                throw new NullPointerException();
328            }
329            // BEGIN android-removed
330            // if (rules.length() == 0) {
331            //     // text.06=Build rules empty
332            //     throw new ParseException(Messages.getString("text.06"), 0); //$NON-NLS-1$
333            // }
334            // END andriod-removed
335    
336            try {
337                this.icuColl = new com.ibm.icu4jni.text.RuleBasedCollator(rules);
338                // BEGIN android-added
339                this.icuColl.setDecomposition(
340                        com.ibm.icu4jni.text.Collator.CANONICAL_DECOMPOSITION);
341                // END android-added
342            } catch (Exception e) {
343                if (e instanceof ParseException) {
344                    throw (ParseException) e;
345                }
346                /*
347                 * -1 means it's not a ParseException. Maybe IOException thrown when
348                 * an error occured while reading internal data.
349                 */
350                throw new ParseException(e.getMessage(), -1);
351            }
352        }
353    
354        /**
355         * Obtains a {@code CollationElementIterator} for the given
356         * {@code CharacterIterator}. The source iterator's integrity will be
357         * preserved since a new copy will be created for use.
358         * 
359         * @param source
360         *            the source character iterator.
361         * @return a {@code CollationElementIterator} for {@code source}.
362         * @since Android 1.0
363         */
364        public CollationElementIterator getCollationElementIterator(
365                CharacterIterator source) {
366            if (source == null) {
367                throw new NullPointerException();
368            }
369            return new CollationElementIterator(
370                    ((com.ibm.icu4jni.text.RuleBasedCollator) this.icuColl)
371                            .getCollationElementIterator(source));
372        }
373    
374        /**
375         * Obtains a {@code CollationElementIterator} for the given string.
376         * 
377         * @param source
378         *            the source string.
379         * @return the {@code CollationElementIterator} for {@code source}.
380         * @since Android 1.0
381         */
382        public CollationElementIterator getCollationElementIterator(String source) {
383            if (source == null) {
384                throw new NullPointerException();
385            }
386            return new CollationElementIterator(
387                    ((com.ibm.icu4jni.text.RuleBasedCollator) this.icuColl)
388                            .getCollationElementIterator(source));
389        }
390    
391        /**
392         * Returns the collation rules of this collator. These {@code rules} can be
393         * fed into the {@link #RuleBasedCollator(String)} constructor.
394         * <p>
395         * Note that the {@code rules} are actually interpreted as a delta to the
396         * standard Unicode Collation Algorithm (UCA). Hence, an empty {@code rules}
397         * string results in the default UCA rules being applied. This differs
398         * slightly from other implementations which work with full {@code rules}
399         * specifications and may result in different behavior.
400         *
401         * @return the collation rules.
402         * @since Android 1.0
403         */
404        public String getRules() {
405            return ((com.ibm.icu4jni.text.RuleBasedCollator) this.icuColl).getRules();
406        }
407    
408        /**
409         * Returns a new collator with the same collation rules, decomposition mode and
410         * strength value as this collator.
411         * 
412         * @return a shallow copy of this collator.
413         * @see java.lang.Cloneable
414         * @since Android 1.0
415         */
416        @Override
417        public Object clone() {
418            RuleBasedCollator clone = (RuleBasedCollator) super.clone();
419            return clone;
420        }
421    
422        /**
423         * Compares the {@code source} text to the {@code target} text according to
424         * the collation rules, strength and decomposition mode for this
425         * {@code RuleBasedCollator}. See the {@code Collator} class description
426         * for an example of use.
427         * <p>
428         * General recommendation: If comparisons are to be done with the same strings
429         * multiple times, it is more efficient to generate {@code CollationKey}
430         * objects for the strings and use
431         * {@code CollationKey.compareTo(CollationKey)} for the comparisons. If each
432         * string is compared to only once, using
433         * {@code RuleBasedCollator.compare(String, String)} has better performance.
434         * </p>
435         * 
436         * @param source
437         *            the source text.
438         * @param target
439         *            the target text.
440         * @return an integer which may be a negative value, zero, or else a
441         *         positive value depending on whether {@code source} is less than,
442         *         equivalent to, or greater than {@code target}.
443         * @since Android 1.0
444         */
445        @Override
446        public int compare(String source, String target) {
447            if (source == null || target == null) {
448                // text.08=one of arguments is null
449                throw new NullPointerException(Messages.getString("text.08")); //$NON-NLS-1$
450            }
451            return this.icuColl.compare(source, target);
452        }
453    
454        /**
455         * Returns the {@code CollationKey} for the given source text.
456         * 
457         * @param source
458         *            the specified source text.
459         * @return the {@code CollationKey} for the given source text.
460         * @since Android 1.0
461         */
462        @Override
463        public CollationKey getCollationKey(String source) {
464            com.ibm.icu4jni.text.CollationKey icuKey = this.icuColl
465                    .getCollationKey(source);
466            if (icuKey == null) {
467                return null;
468            }
469            return new CollationKey(source, icuKey);
470        }
471    
472        @Override
473        public int hashCode() {
474            return ((com.ibm.icu4jni.text.RuleBasedCollator) this.icuColl).getRules()
475                    .hashCode();
476        }

478        /**
479         * Compares the specified object with this {@code RuleBasedCollator} and
480         * indicates if they are equal. In order to be equal, {@code object} must be
481         * an instance of {@code Collator} with the same collation rules and the
482         * same attributes.
483         * 
484         * @param obj
485         *            the object to compare with this object.
486         * @return {@code true} if the specified object is equal to this
487         *         {@code RuleBasedCollator}; {@code false} otherwise.
488         * @see #hashCode
489         * @since Android 1.0
490         */
491        @Override
492        public boolean equals(Object obj) {
493            if (!(obj instanceof Collator)) {
494                return false;
495            }
496            return super.equals(obj);
497        }
